# 防洪调度智能问答系统 — 需求文档

**文档版本**：v1.0  
**适用系统**：基于 RAG + 知识图谱 + Agent 的防洪调度智能问答系统（PaiSmart）

---

## 一、业务背景与目标

### 1.1 业务场景

- 面向**水利/防洪调度操作**的智能问答，为相关部门提供规程查询、站点关系、汛期调度等决策支持。
- 支持多格式文档上传与解析，结合语义检索与大模型，针对具体场景给出解决方案。

### 1.2 目标用户

- 调度人员、管理人员等；需**多级权限管控**，保障敏感信息与规程安全。

### 1.3 系统目标

- 用户用自然语言提问，系统能**解析意图**、**按需查知识库与知识图谱**、**多步推理后**给出准确、可溯源的回答。
- 通过流式对话与上下文记忆提升操作效率。

---

## 二、核心能力需求

### 2.1 意图识别

- **作用**：判断用户问题属于哪一类，从而决定用哪些数据源、是否走多步推理。
- **需求**：
  - 能区分：**纯文档/规程查询**（走 RAG）、**纯实体关系查询**（走 KG）、**既查关系又查规程**（走 RAG+KG 或 Agent）。
  - 能识别**多步推理类**问题（如“某站点**下游测站**的**汛期调度规程**”），这类问题应走 Agent。
  - 能从问句中抽取**槽位**：河流/河段、站点名、关系类型（下游/上游/测站）、时间（汛期）等，供 KG 与 RAG 使用。
- **可选扩展**：先规则/关键词，后续可升级为小模型或 LLM 做意图分类与槽位抽取。

### 2.2 知识库（RAG）

- **内容**：规程、办法、文档等非结构化文本，经解析、切分、向量化后存入 ES，支持语义+关键词混合检索。
- **需求**：
  - 支持按**查询词**检索（如“汛期调度规程”“某站点规程”），返回相关条文/片段。
  - 检索结果需带**权限过滤**（用户本人、公开、所属组织），并标注来源（文件名、片段）。
  - 大模型回答时能**引用来源**（如“依据规程第 X 条”），避免跨记录拼接（可结合元数据摘要）。

### 2.3 知识图谱（KG）

- **内容**：河流、河段、站点、闸门等**实体**及其**关系**（上下游、隶属、控制关系等）。
- **需求**：
  - 支持按**实体+关系类型**查询，例如：“淮沭新河-泗洪段-xx站点”的“下游测站”有哪些。
  - 返回结果可为**三元组或自然语言描述**，供大模型理解后再决定是否查 RAG 或站点属性。
  - 与 RAG 配合：先由 KG 得到“下游测站列表”，再按站点名+“汛期规程”去 RAG 检索。

### 2.4 多步推理（Agent）

- **触发**：当问题同时涉及**实体关系**（下游/上游/测站等）和**规程/文档**（调度规程、汛期办法等）时，走 Agent 多步推理。
- **典型问题**：  
  “淮沭新河在泗洪段的 xx 站点的**下游测站点**的**汛期**时候的**调度规程**是什么？”
- **预期流程**：
  1. **步骤 1**：理解问题，确定要查“淮沭新河-泗洪段-xx站点”的“下游测站”。
  2. **步骤 2**：调用 **KG 工具**，得到下游测站列表（或子图描述）。
  3. **步骤 3**：根据下游测站名称 + “汛期”“调度规程”等，调用 **RAG 工具**检索条文；或调用**站点属性工具**查该站点的规程引用。
  4. **步骤 4**：综合 KG 与 RAG 结果，生成 **FINAL_ANSWER**，并注明依据来源。
- **需求**：
  - Agent 具备**工具集**：至少包括 KG 查询、RAG 检索、站点属性（可先占位）。
  - 由 LLM 决定**调用顺序与参数**（河段、站点、关系类型、检索词等）。
  - 步数上限、超时与错误处理可配置，避免死循环或长时间阻塞。

### 2.5 回答呈现与溯源

- **需求**：
  - 回答需**先结论、再论据**，并尽量标注**来源**（知识库文档名/条款、知识图谱关系）。
  - 支持**流式输出**（单轮 RAG+KG 可流式；Agent 多步可在全部步骤完成后对最终答案做流式或一次性返回）。
  - 对话**历史**可留存，支持多轮追问（上下文记忆）。

## 三、知识引擎模块需求（大量 API）

知识引擎模块对外提供**统一的知识能力 API**，供前端、Agent、其他微服务等调用。以下为能力分类与接口清单。

### 3.1 检索类 API

| 接口说明 | 方法 | 路径（示例） | 说明 |
|----------|------|--------------|------|
| 混合检索（带权限） | GET | `/api/v1/knowledge/search/hybrid` | 向量+BM25 混合检索，按 userId 做权限过滤 |
| 混合检索（指定 topK） | GET | `/api/v1/knowledge/search/hybrid?query=xxx&topK=10` | 同上，可传 topK |
| 按文件查 Chunk 列表 | GET | `/api/v1/knowledge/documents/{fileMd5}/chunks` | 返回该文件下所有文本块（需权限） |
| 单 Chunk 详情 | GET | `/api/v1/knowledge/documents/{fileMd5}/chunks/{chunkId}` | 返回指定 chunk 的文本内容 |

### 3.2 知识图谱类 API

| 接口说明 | 方法 | 路径（示例） | 说明 |
|----------|------|--------------|------|
| 子图/关系查询 | GET | `/api/v1/knowledge/kg/subgraph?query=xxx&river=xx&segment=xx&site=xx&relation=下游测站` | 按 query + 槽位查图谱，返回文本描述或三元组 |
| 子图/关系查询（Body） | POST | `/api/v1/knowledge/kg/subgraph` | Body: `{"query":"xxx","slots":{"river":"淮沭新河","relation":"下游测站"}}` |
| 实体查询（扩展） | GET | `/api/v1/knowledge/kg/entity?name=xx站点` | 查询实体属性（可选实现） |
| 关系列表（扩展） | GET | `/api/v1/knowledge/kg/relations?from=站点A&type=下游` | 查询某实体的某类关系（可选实现） |

### 3.3 文档/元数据类 API

| 接口说明 | 方法 | 路径（示例） | 说明 |
|----------|------|--------------|------|
| 当前用户可访问文件列表 | GET | `/api/v1/knowledge/documents/accessible` | 与现有文档列表一致，可委托文档服务 |
| 文件摘要信息 | GET | `/api/v1/knowledge/documents/{fileMd5}/summary` | 文件名、chunk 数、上传时间等（可选） |

### 3.4 统计与健康类 API

| 接口说明 | 方法 | 路径（示例） | 说明 |
|----------|------|--------------|------|
| 索引/文档统计 | GET | `/api/v1/knowledge/stats` | 知识库文档数、chunk 数、索引状态等 |
| 引擎健康检查 | GET | `/api/v1/knowledge/health` | ES、KG 等依赖是否可用 |

### 3.5 实现对应关系

- 检索类：由 **KnowledgeEngineService** 门面委托 **HybridSearchService**、**DocumentVectorRepository** 等实现。
- 知识图谱类：由 **KnowledgeGraphService** 实现，当前可为占位，后续对接 Neo4j 等图库。
- 上述 API 路径为建议，实际以项目内 **KnowledgeEngineController**（或等价控制器）为准；权限需与现有 RBAC/组织标签一致。

## 四、典型问题与预期行为（示例）

| 用户问题 | 意图 | 数据源 | 预期行为 |
|----------|------|--------|----------|
| 汛期调度规程里对水位有什么规定？ | 规程条文 | RAG | 混合检索“汛期”“水位”“规程”，返回相关条文并引用来源。 |
| 淮沭新河泗洪段有哪些下游闸门？ | 实体关系 | KG | 查 KG：河段+站点+关系“下游闸门”，返回实体列表或子图描述。 |
| 淮沭新河在泗洪段的 xx 站点的下游测站点的汛期调度规程是什么？ | 多步：关系+规程 | Agent（KG→RAG） | 先 KG 查“xx站点”的“下游测站”，再 RAG 查“下游测站名+汛期+调度规程”，综合成答并注明来源。 |

## 五、非功能需求

- **权限**：知识库与图谱访问需与现有 RBAC/组织标签一致，Agent 内 RAG 调用带用户身份。
- **性能**：单轮 RAG+KG 响应时间可接受；Agent 多步可设步数上限（如 5–6 步）与超时。
- **可扩展**：意图类型、Agent 工具（如新增“查实时水位”）、KG 关系类型、知识引擎 API 均可后续扩展，不绑死在一版实现上。

---

## 六、术语与边界

- **知识库**：指文档/规程经解析、切分、向量化后存入的检索系统（当前为 ES），即 RAG 的数据源。
- **知识图谱**：指实体与关系组成的图结构数据，用于“下游测站”“上游闸门”等查询，可与 RAG 联合使用。
- **知识引擎模块**：对外提供“大量 API”的模块，统一封装检索、图谱、文档 chunk、统计与健康等能力。
- **Agent**：在本需求中特指“多步推理 + 工具调用”的流程，由 LLM 决定何时调 KG、何时调 RAG、何时给出最终答案；不要求必须用某一具体框架名（如 ReAct），只要行为符合上述流程即可。

## 七、与现有实现的对应关系

| 需求项 | 当前实现/可扩展点 |
|--------|-------------------|
| 意图识别 | `IntentRecognitionService`（规则/关键词），可扩展为模型 |
| RAG | `HybridSearchService` + ES + 权限过滤，已支持 |
| 知识图谱 | `KnowledgeGraphService` 接口 + 占位实现，待接入真实图库 |
| Agent 多步 | `AgentService` + `AgentTool`（query_kg, search_rag, get_site_attributes），待在 ChatHandler 中按“多步推理类”意图触发 |
| 回答溯源 | system 中区分【知识库检索结果】与【知识图谱】，模型按说明引用 |
| 知识引擎 API | `KnowledgeEngineService` 门面 + `KnowledgeEngineController`（可按第三节清单逐步实现） |
