# 防洪调度智能问答系统 — 需求说明

## 一、业务背景与目标

- **场景**：面向水利/防洪调度操作的智能问答，为相关部门提供规程查询、站点关系、汛期调度等决策支持。
- **用户**：调度人员、管理人员等，需多级权限管控，保障敏感信息与规程安全。
- **目标**：用户用自然语言提问，系统能**解析意图**、**按需查知识库与知识图谱**、**多步推理后**给出准确、可溯源的回答。

---

## 二、核心能力需求

### 2.1 意图识别

- **作用**：判断用户问题属于哪一类，从而决定用哪些数据源、是否走多步推理。
- **需求**：
  - 能区分：**纯文档/规程查询**（走 RAG）、**纯实体关系查询**（走 KG）、**既查关系又查规程**（走 RAG+KG 或 Agent）。
  - 能识别**多步推理类**问题（如“某站点**下游测站**的**汛期调度规程**”），这类问题应走 Agent。
  - 能从问句中抽取**槽位**：河流/河段、站点名、关系类型（下游/上游/测站）、时间（汛期）等，供 KG 与 RAG 使用。
- **可选扩展**：先规则/关键词，后续可升级为小模型或 LLM 做意图分类与槽位抽取。

### 2.2 知识库（RAG）

- **内容**：规程、办法、文档等非结构化文本，经解析、切分、向量化后存入 ES，支持语义+关键词混合检索。
- **需求**：
  - 支持按**查询词**检索（如“汛期调度规程”“某站点规程”），返回相关条文/片段。
  - 检索结果需带**权限过滤**（用户本人、公开、所属组织），并标注来源（文件名、片段）。
  - 大模型回答时能**引用来源**（如“依据规程第 X 条”），避免跨记录拼接（可结合元数据摘要）。

### 2.3 知识图谱（KG）

- **内容**：河流、河段、站点、闸门等**实体**及其**关系**（上下游、隶属、控制关系等）。
- **需求**：
  - 支持按**实体+关系类型**查询，例如：“淮沭新河-泗洪段-xx站点”的“下游测站”有哪些。
  - 返回结果可为**三元组或自然语言描述**，供大模型理解后再决定是否查 RAG 或站点属性。
  - 与 RAG 配合：先由 KG 得到“下游测站列表”，再按站点名+“汛期规程”去 RAG 检索。

### 2.4 多步推理（Agent）

- **触发**：当问题同时涉及**实体关系**（下游/上游/测站等）和**规程/文档**（调度规程、汛期办法等）时，走 Agent 多步推理。
- **典型问题**：  
  “淮沭新河在泗洪段的 xx 站点的**下游测站点**的**汛期**时候的**调度规程**是什么？”
- **预期流程**：
  1. **步骤 1**：理解问题，确定要查“淮沭新河-泗洪段-xx站点”的“下游测站”。
  2. **步骤 2**：调用 **KG 工具**，得到下游测站列表（或子图描述）。
  3. **步骤 3**：根据下游测站名称 + “汛期”“调度规程”等，调用 **RAG 工具**检索条文；或调用**站点属性工具**查该站点的规程引用。
  4. **步骤 4**：综合 KG 与 RAG 结果，生成 **FINAL_ANSWER**，并注明依据来源。
- **需求**：
  - Agent 具备**工具集**：至少包括 KG 查询、RAG 检索、站点属性（可先占位）。
  - 由 LLM 决定**调用顺序与参数**（河段、站点、关系类型、检索词等）。
  - 步数上限、超时与错误处理可配置，避免死循环或长时间阻塞。

### 2.5 回答呈现与溯源

- **需求**：
  - 回答需**先结论、再论据**，并尽量标注**来源**（知识库文档名/条款、知识图谱关系）。
  - 支持**流式输出**（单轮 RAG+KG 可流式；Agent 多步可在全部步骤完成后对最终答案做流式或一次性返回）。
  - 对话**历史**可留存，支持多轮追问（上下文记忆）。

## 三、典型问题与预期行为（示例）

| 用户问题 | 意图 | 数据源 | 预期行为 |
|----------|------|--------|----------|
| 汛期调度规程里对水位有什么规定？ | 规程条文 | RAG | 混合检索“汛期”“水位”“规程”，返回相关条文并引用来源。 |
| 淮沭新河泗洪段有哪些下游闸门？ | 实体关系 | KG | 查 KG：河段+站点+关系“下游闸门”，返回实体列表或子图描述。 |
| 淮沭新河在泗洪段的 xx 站点的下游测站点的汛期调度规程是什么？ | 多步：关系+规程 | Agent（KG→RAG） | 先 KG 查“xx站点”的“下游测站”，再 RAG 查“下游测站名+汛期+调度规程”，综合成答并注明来源。 |

## 四、非功能需求

- **权限**：知识库与图谱访问需与现有 RBAC/组织标签一致，Agent 内 RAG 调用带用户身份。
- **性能**：单轮 RAG+KG 响应时间可接受；Agent 多步可设步数上限（如 5–6 步）与超时。
- **可扩展**：意图类型、Agent 工具（如新增“查实时水位”）、KG 关系类型均可后续扩展，不绑死在一版实现上。

---

## 五、术语与边界

- **知识库**：指文档/规程经解析、切分、向量化后存入的检索系统（当前为 ES），即 RAG 的数据源。
- **知识图谱**：指实体与关系组成的图结构数据，用于“下游测站”“上游闸门”等查询，可与 RAG 联合使用。
- **Agent**：在本需求中特指“多步推理 + 工具调用”的流程，由 LLM 决定何时调 KG、何时调 RAG、何时给出最终答案；不要求必须用某一具体框架名（如 ReAct），只要行为符合上述流程即可。

## 六、与现有实现的对应关系

| 需求项 | 当前实现/可扩展点 |
|--------|-------------------|
| 意图识别 | `IntentRecognitionService`（规则/关键词），可扩展为模型 |
| RAG | `HybridSearchService` + ES + 权限过滤，已支持 |
| 知识图谱 | `KnowledgeGraphService` 接口 + 占位实现，待接入真实图库 |
| Agent 多步 | `AgentService` + `AgentTool`（query_kg, search_rag, get_site_attributes），待在 ChatHandler 中按“多步推理类”意图触发 |
| 回答溯源 | system 中区分【知识库检索结果】与【知识图谱】，模型按说明引用 |

此文档可作为开发与评审的基准，后续具体设计（如 KG 建模、Agent 触发条件）可在此需求上细化。
