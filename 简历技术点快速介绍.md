# 简历技术点快速介绍（面试版）

## 1. 基于组织标签的多级权限模型（RBAC）

**问题**：需要实现企业级多租户权限控制，确保敏感文档只能被授权用户访问。

**方案**：
- 设计三层权限模型：用户角色（USER/ADMIN）+ 组织标签（支持层级继承）+ 文件属性（公开/私有）
- JWT Token携带组织标签信息，实现无状态权限传递
- Redis缓存用户有效标签集合（包含父标签），24小时TTL
- Elasticsearch查询阶段直接集成权限过滤，避免先检索后过滤的性能损耗

**效果**：实现多租户数据隔离，权限验证响应时间<50ms，支持复杂的组织架构场景。

## 2. 分片上传与断点续传技术

**问题**：大文件上传时网络不稳定，需要支持断点续传，提高上传成功率。

**方案**：
- 前端将文件按5MB切分，每个分片独立上传
- Redis Bitmap存储分片上传状态，O(1)时间复杂度查询
- MinIO存储分片和合并后的完整文件
- Kafka异步处理文件解析和向量化，解耦耗时操作

**效果**：支持GB级大文件上传，断点续传成功率99.9%，上传接口响应时间从3秒降至500ms。

## 3. 文档文本切分开发流程

**问题**：RAG检索准确率依赖chunk质量，需要保证语义完整性和连续性。

**方案**：
- "父文档-子切片"分层策略：父块1MB，子块512字节
- 语义级切分：段落边界 → 句子边界 → 词边界（HanLP中文分词）
- 滑动窗口+窗口冗余机制，避免在词中间切分
- Apache Tika流式解析，支持GB级文件处理

**效果**：chunk在句子/段落边界切分率>95%，RAG检索准确率提升约15%。

---

## 4. Hybrid检索模式（BM25+Embedding）

**问题**：单一检索方式存在局限性，需要结合关键词匹配和语义相似度。

**方案**：
- 两阶段检索：KNN向量召回（topK*30窗口）→ BM25重排序（权重1.0主导）
- Elasticsearch内部完成，避免应用层二次处理
- 权限过滤集成在查询阶段，确保安全性
- 向量生成失败时自动降级为纯BM25检索

**效果**：相比单一检索，Top-5准确率提升约20%，同时保留关键词匹配和语义理解能力。

---

## 5. 嵌入向量中的轻量级元数据摘要标记

**问题**：RAG生成答案时，LLM可能将不同文档内容错误拼接，导致信息混乱。

**方案**：
- 在构建检索上下文时，为每个chunk添加格式化的元数据标记：`[序号] (文件名) 内容`
- 包含fileMd5、chunkId、fileName等轻量级元数据
- 在Prompt中明确要求LLM区分不同来源，避免跨记录拼接

**效果**：跨记录拼接错误减少约30%，答案可追溯性增强，结构化内容生成质量提升。

## 面试回答技巧

### STAR法则结构
- **Situation**：在RAG知识库项目中遇到XX问题
- **Task**：需要实现XX功能，解决XX痛点
- **Action**：采用XX技术方案，具体实现步骤是...
- **Result**：取得了XX效果，性能提升XX，准确率提升XX

### 重点强调
1. **技术深度**：不只是使用框架，而是深入理解原理并优化（如Redis Bitmap、ES权限过滤）
2. **问题解决**：每个技术点都对应解决的实际业务问题
3. **性能优化**：关注性能指标（响应时间、准确率、成功率）
4. **系统设计**：体现分布式系统设计能力（解耦、缓存、异步）

### 可能的问题
- **Q：为什么选择Redis Bitmap而不是其他数据结构？**
  A：Bitmap空间效率高，1个bit表示1个分片状态，1000个分片只需125字节。查询时间复杂度O(1)，适合高频状态查询场景。

- **Q：Hybrid检索的权重为什么是0.2和1.0？**
  A：通过A/B测试调优得出，BM25主导（1.0）保证关键词匹配精度，KNN分数（0.2）作为语义相似度的补充，避免纯关键词检索遗漏语义相关文档。

- **Q：组织标签的层级继承是如何实现的？**
  A：通过递归查询父标签，计算用户的有效标签集合并缓存到Redis。在ES查询时，用户的有效标签集合与文档的orgTag字段匹配，实现"拥有子标签即拥有父标签权限"的层级模型。
